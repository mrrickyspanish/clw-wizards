# Closeout Report Template

Version: 2.0

Use this report after completing meaningful work for RJ. The seven headings are canonical (see `core-rules.md`, section 4). This file adds guidance and examples for each.

Keep it direct. No filler. No fake certainty.

## Outcome

State what was completed in plain English.

Example:

The supporting service pages were refined for clearer copy, stronger CTAs, and basic SEO metadata consistency.

## What Changed

Group the changes by category.

Use only relevant categories:

### Copy

What changed in headlines, body copy, CTAs, tone, clarity, or messaging.

### Conversion

What changed in CTA flow, trust cues, inquiry path, page purpose, or friction reduction.

### SEO

What changed in page titles, meta descriptions, H1s, H2s, internal links, alt text, or search intent.

### UX and Readability

What changed in mobile readability, spacing, hierarchy, card density, or visual flow.

### Design

What changed visually, if applicable.

### Code or Technical

What changed in components, files, logic, routing, data, styles, or build behavior.

### Cleanup

Any removed duplication, corrected language, simplified structure, or scope cleanup.

## Why It Matters

Explain the practical business value.

Examples:

- Improves visitor clarity
- Makes the next step easier
- Strengthens trust
- Improves mobile readability
- Supports search visibility
- Keeps the brand from sounding generic
- Reduces production risk
- Makes future execution easier

## Files Touched or Deliverables Created

List every file touched or every artifact created.

Do not hide files.

## Verification

State exactly what was checked.

Include:

- Build status
- Lint status
- Type check status
- Visual review status
- Mobile review status
- Metadata review status
- Link review status
- Any skipped checks and why

## Open Risks

List anything uncertain, incomplete, not checked, or dependent on RJ/client review.

Examples:

- Visual review limited to local preview
- Build could not be run
- CTA language needs final taste call
- Client proof points were not available
- SEO keyword target was inferred

## RJ Taste Calls

List only the decisions that genuinely need RJ's review.

Do not include routine execution choices that were already within scope.

Good taste calls:

- Whether the headline feels too direct
- Whether a visual direction matches the brand
- Whether a pricing framing should be used
- Whether to add or remove a section

Bad taste calls:

- "Please approve these minor typo fixes"
- "Let me know if you like it" with no specific decision
- Asking RJ to approve work that was clearly requested
