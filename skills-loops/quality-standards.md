# Quality Standards

Version: 2.0

## Purpose

These are RJ's quality standards for AI-assisted work across brands, clients, content, code, and creative direction.

Every output should be practical, scoped, and useful.

## Strategy standard

Good strategy work should:

- Identify the business reason
- Clarify the audience
- Define the actual problem
- Separate taste from objective function
- Identify risks and assumptions
- Recommend a clear next move
- Avoid generic frameworks unless they help
- Challenge weak thinking directly
- Protect long-term positioning

Bad strategy work:

- Agrees with everything
- Uses buzzwords
- Creates fake certainty
- Ignores second-order risks
- Gives long lists instead of a recommendation
- Treats emotional truth as automatic public strategy
- Misses what people will assume

## Copy standard

Good copy should:

- Sound human
- Be specific
- Be easy to scan
- Make the next step obvious
- Fit the brand and audience
- Explain business value
- Avoid filler
- Avoid over-polish
- Avoid AI tells
- Use strong headlines with clear meaning

Bad copy:

- Sounds like a generic agency
- Uses vague superlatives
- Says "elevate," "transform," or "unlock" without substance
- Makes claims without proof
- Over-explains simple things
- Feels disconnected from the audience
- Uses em dashes in copy meant for RJ to publish

## Conversion standard

Good conversion work should:

- Clarify what the visitor should do next
- Reduce hesitation
- Strengthen trust
- Place CTAs where they make sense
- Connect services to business outcomes
- Make contact or inquiry feel low-friction
- Use proof, process, specificity, or examples when available
- Avoid fake urgency

Bad conversion work:

- Adds loud CTAs without logic
- Pushes too hard too early
- Makes the page longer without making it clearer
- Adds sections just because competitors have them
- Uses vague "book now" language with no context
- Ignores the visitor's actual concern

## SEO standard

Good SEO work should:

- Align with natural search intent
- Use unique page titles
- Use useful meta descriptions
- Keep H1 and H2 structure logical
- Use descriptive internal links
- Improve alt text where meaningful
- Use local relevance only when natural
- Avoid keyword stuffing
- Support human clarity first

Bad SEO work:

- Forces keywords everywhere
- Creates awkward location stuffing
- Writes metadata that sounds robotic
- Adds thin content just to add words
- Ignores conversion
- Makes pages feel bloated

## UX readability standard

Good UX readability should:

- Preserve hierarchy
- Keep body copy readable, especially on mobile
- Use reasonable line height
- Keep cards scannable
- Avoid cramped sections
- Avoid tiny 14px or 15px important body copy unless there is a clear reason
- Improve flow without inflating the design
- Make the page feel easier to understand

Bad UX readability:

- Hides important information in tiny text
- Creates dense paragraphs
- Uses too many competing styles
- Overuses all caps
- Makes mobile feel like an afterthought
- Prioritizes decoration over comprehension

## Visual design standard

Good visual design should:

- Feel intentional
- Match the brand's actual context
- Use strong composition
- Have clear focal points
- Respect the reference
- Support the message
- Avoid generic AI polish
- Avoid visual clutter
- Use color and texture with purpose

Bad visual design:

- Looks like fantasy art when a logo was requested
- Adds random effects
- Ignores aspect ratio
- Adds fake text
- Uses stock-photo energy
- Overloads the frame
- Treats mood as more important than utility

## Image prompt standard

Full rules live in `creative-output-rules.md`. Summary:

For high-impact image prompts, use structured sections:

- Subject
- Environment
- Composition
- Lighting
- Energy or Motion Suggestion
- Texture and Detail Level
- Color Strategy
- Rendering Style

Internally clarify:

- Core moment
- Emotional tone
- Brand implication
- What must be avoided
- Camera angle rationale
- Lighting rationale
- Color strategy
- Composition intent and power dynamic

Do not output internal analysis unless RJ asks for it.

## Coding standard

Good code work should:

- Be scoped
- Preserve existing architecture
- Avoid unrelated refactors
- Keep changes readable
- Build successfully
- Respect branch workflow
- Include clear commit messages when committing
- Report files touched
- Explain what was verified
- Acknowledge what was not verified

Bad code work:

- Changes unrelated pages
- Breaks build
- Hides errors
- Adds unnecessary dependencies
- Rewrites architecture without reason
- Changes production without instruction
- Makes broad "cleanup" changes outside scope

## Git standard

Use Git discipline:

- Create a fresh branch for meaningful tasks.
- Use clear branch names.
- Keep commits focused.
- Do not merge without instruction.
- Do not promote to production without instruction.
- Do not delete unrelated work.
- Report branch name and commit message.

## Client-facing deliverable standard

Good client-facing work should:

- Be polished but plain
- Make value clear
- Tie deliverables to outcomes
- Avoid inflated language
- Use clean hierarchy
- Be easy to review
- Avoid accidental overpromising
- Keep scope and pricing clear when included

## Internal brief standard

Good internal briefs should:

- Define the mission
- Explain the business reason
- Provide inputs
- Set deliverables
- Set guardrails
- Define pause conditions
- Define verification
- State the final report format
- Prevent overbuilding

## Final report standard

A good final report should answer:

- What happened?
- What changed?
- Why does it matter?
- What files or deliverables were touched?
- What was verified?
- What risks remain?
- What does RJ actually need to decide?

A bad final report:

- Uses vague praise
- Hides uncertainty
- Omits files changed
- Says "done" without verification
- Asks RJ to approve things already in scope
